## 2026-05-11: 진단 모듈 spec v0.1 확정

**결정 사항**:
- 4 Layer 구조 → 2 Layer로 축소 (Layer 3, 4는 27년 봄 이후)
- Layer 2 sub-category 5개 (인력 안정성 / 채용 / 총보상 / 평가 / 리더십 거버넌스)
- 총 26개 변수, 입력 시간 35~45분
- 가시성 지수 = Product 시그니처 IP
- 8주 deploy 워크플로우에 Week 2-3 사각지대 채우기 단계 포함

## 2026-05-12: SSOT 변수 구현 기준 확정

**컨텍스트**: 진단 문항의 논리적 묶음은 26개지만, Streamlit UI와 시뮬레이션 계산을 위해 일부 세부 입력을 별도 변수로 분리할 필요가 있음.

**결정 사항**:
- `ALL_VARIABLES`는 UI 구현 단위 기준 30개로 유지
- `2-3-2-detail`, `2-4-3-ceo`, `2-4-3-employee` 등 세부 입력 ID는 별도 변수로 관리
- 체크리스트의 변수 개수 기대값은 26개가 아니라 30개로 해석

**근거**: 논리 문항 수와 UI/계산 단위의 변수 수를 분리해야 Layer 2 폼, 가시성 지수, 시뮬레이션 계산에서 상태 관리가 단순해짐.

## 2026-05-12: Layer 2 + 가시성 지수 + 진단 결과 페이지 완성

**컨텍스트**: #03 작업 지시서 기준으로 진단 모듈을 end-to-end로 완성해야 함. Layer 2 입력, 조건부 문항, 가시성 지수 계산, 결과 페이지, SQLite 저장 흐름이 하나의 진단 세션으로 이어져야 함.

**결정 사항**:
- Layer 2는 5개 sub-category expander를 모두 기본 펼침 상태로 렌더링
- 단일 선택 문항은 `선택해 주세요` placeholder를 둬서 응답이 자동 선택되지 않도록 처리
- `2-4-1 = 없음`이면 `2-4-5`를 숨기고 가시성 지수 분모에서 제외
- `2-5-2 = 운영 안 함`이면 `2-5-3`을 숨기고 가시성 지수 분모에서 제외
- `2-3-2-detail`은 선택 입력으로 두고, 미입력 시 보상 구조 가시성을 0.5점으로 계산
- 진단 결과 페이지에는 Visibility Index 게이지, tier 메시지, 사각지대 목록을 표시
- Layer 1 → Layer 2 → Result 이동 시 SQLite 세션을 저장하고 current_step을 갱신

**근거**: CEO 입력 부담을 낮추면서도 "무엇을 모르는지"를 결과 화면에 명확히 드러내는 것이 Transition Gap의 시그니처 IP와 HITL 워크플로우에 부합함.

## 2026-05-11: 시뮬레이션 모듈 — 매트릭스 사분면 명명 확정

**매트릭스 A (Motivation Mix)**:
- Q1 단기 성과형 용병조직
- Q2 장기 비전형 공동체 조직
- Q3 평균의 함정형
- Q4 소수정예 중심형

**매트릭스 B (Operating System)**:
- Q1 개인플레이어 중심형
- Q2 가족형 자율 조직
- Q3 에이전시형 분업/기능 조직
- Q4 대기업 공채 시스템형

**풍선 효과(Balloon Effect) 프레임**: 트레이드오프 양 끝점은 절충 가능하지만, 절충은 막대한 관리 비용을 요구한다. 진단 결과(리더십 역량 등)에 따라 절충 가능성이 결정된다는 메시지를 유지한다.

## 2026-05-15: 변수 구조 v0.2 재정렬

**컨텍스트**: 시뮬레이션 매핑 정밀도를 높이기 위해 페인포인트, 평가 방식, 핵심 인재 이탈, 인원 규모 변수를 동시에 조정해야 함.

**결정 사항**:
- L1-1 페인포인트를 5개에서 8개로 교체하고 매트릭스 A Y축 매핑을 갱신
- 기존 `2-4-1` 평가 방식 단일 문항을 `2-4-1a/b/c/d` 4개 문항으로 분리
- `2-4-5` 조건부 표시/가시성 분모 기준을 `2-4-1a != 운영하지 않음`으로 변경
- L1-2 인원수에 `100~500인`, `500인 초과` 구간을 반영
- 핵심 인재 이탈(`2-1-2`)은 인원 규모와 교차하는 비선형 심각도 함수로 별도 계산

**근거**: CEO 단독 입력 구조를 유지하면서도 실제 제도 설계에서 중요한 맥락 차이를 시뮬레이션 계산에 더 정확히 반영하기 위함.

## 2026-05-15: Layer 2 화면 3분할 및 사이드바 이동 구조

**근거**: CEO 입력 흐름의 인지 부하를 낮추면서도 미입력 검증과 데이터 저장 흐름을 깨지 않기 위함.

## 2026-05-15: 응답 데이터 이중 안전망 적용

**컨텍스트**: Streamlit rerun/새로고침 과정에서 `st.session_state.responses` 의존만으로는 입력 데이터가 날아갈 위험이 남아 있음.

- 앱 시작 시 가장 최근 세션의 응답 데이터만 복원하고, 진행 단계는 `layer1` 흐름을 유지
- 5점 입력은 슬라이더 대신 가로 라디오로 표시하되 DB 저장값은 1~5 정수로 유지

**근거**: 입력 경험을 가볍게 만들면서도 튕김/새로고침 상황에서 데이터 소실을 최소화하기 위함.

## 2026-05-18: 랜딩 복구 및 진단 UX 폴리시

**컨텍스트**: #04.10 화면 분할 과정에서 시작 안내가 빠지고, 단계 이동 버튼과 5점 입력 UX에 남은 마찰이 확인됨.

**결정 사항**:

- 5점 입력은 가로 라디오를 유지하되 양 끝점 스펙트럼 힌트를 제공

**근거**: 입력자의 부담을 낮추면서도 결과 페이지 접근 전 필수 데이터 완성도를 유지하기 위함.

## 2026-05-18: 5점 스펙트럼 입력 및 노출 브랜딩 정리

**컨텍스트**: #04.11에서 5점 스펙트럼 문항을 가로 라디오로 바꾼 뒤 디자인 검토에서 슬라이더 방식이 더 적합하다고 판단됨. 동시에 사용자 노출 카피에서 도구가 단순 자동화처럼 보이지 않도록 표현을 정리할 필요가 있음.

**결정 사항**:
- `SLIDER_5` 문항 5개는 `st.select_slider`로 복원하고, DB 저장값은 기존과 동일하게 1~5 정수로 유지
- 각 5점 문항 위에는 좌우 끝점 스펙트럼 라벨을 표시
- 사용자 노출 카피는 "AI" 전면 브랜딩 대신 정합성 진단, 전문가 검토, HITL 표현을 사용
- 산업 도메인 선택지의 노출 라벨도 "데이터 / 자동화"로 조정

**근거**: 데이터 구조와 계산 로직을 유지하면서도 CEO 입력 경험을 스펙트럼 선택에 가깝게 만들고, B2B 컨설팅 도구 톤을 강화하기 위함.

## 2026-05-18: 영역별 상세 분석 엔진 도입

**결정 사항**:
- 평가 공정성 인식 문항 2개(`2-4-3-ceo`, `2-4-3-employee`)를 10점 척도로 확장
- 보상, 평가, 채용, 인력 안정성, 리더십·거버넌스 5개 영역의 점수·갭·이슈·추천 방향을 생성하는 `analysis_engine.py` 추가
- 제도 옵션과 벤치마크는 `src/data/hr_options.py` 정적 콘텐츠로 분리
- 결과 페이지에는 교차 인사이트, 영역별 갭 테이블, 영역별 5단 상세 분석을 추가

**근거**: 단순 진단 점수보다 "어디를 먼저, 어떤 순서로 다룰지"를 제시해야 Transition Gap의 HITL 컨설팅 도구 정체성이 강화됨.

## 2026-05-18: 갭 차트 및 시나리오 상세 카드 도입

**컨텍스트**: #05.3 결과 페이지의 영역별 갭 비교가 표 중심이라 우선순위와 목표 대비 차이가 즉시 읽히지 않았고, 시뮬레이션 페이지는 매트릭스 이후의 선택 가능한 전략 패키지가 부족했음.

**결정 사항**:

- 차트 하단에 우선순위와 난이도 배지를 추가
- 시뮬레이션 페이지에는 성과주의 가속형, 공동체 안정형, 소수정예 집중형 3개 시나리오 탭을 추가
- 시나리오 데이터는 `src/data/scenarios.py`, 렌더링은 `src/diagnosis/scenario_renderer.py`로 분리

**근거**: 진단 결과에서 개선 우선순위를 더 빠르게 이해하고, 매트릭스 이후 구체적인 실행 선택지를 비교할 수 있게 하기 위함.

## 2026-05-18: 실행 로드맵과 To-Be 매트릭스 표시 도입

**컨텍스트**: 시나리오 상세 카드가 전략 패키지를 비교하는 데는 충분하지만, 선택 이후 "무엇을 어떤 순서로 실행할지"가 별도 흐름으로 드러나지 않았음.

**결정 사항**:
- 시뮬레이션 페이지에 실행 시나리오 선택 라디오와 4단계 실행 로드맵 타임라인을 추가
- 로드맵은 `SCENARIOS[*]["next_steps"]`를 SSOT로 사용하고, 세부 패키지는 expander로 제공
- 매트릭스 A/B에는 사분면 배경색, 축 방향 표시, 선택 시나리오 기반 To-Be 기준점, As-Is→To-Be 벡터를 렌더링
- 재무 임팩트 색상은 금액 문자열 부호가 아니라 `color_intent` 메타데이터로 판단

**근거**: 사용자가 전략 비교에서 끝나지 않고 실행 순서를 즉시 이해해야 HITL 워크샵과 8주 deploy 흐름으로 자연스럽게 이어질 수 있음.

## 2026-05-18: 결과 페이지 분할 및 점수 설명 전면화

- 결과 요약의 첫 화면은 점수표나 레이더 차트보다 “제도들이 같은 방향을 보는가”를 보여주는 정합성 맵을 우선한다.

**근거**: 시뮬레이터가 먼저 나오면 기능은 화려하지만 왜 보상을 바꿔야 하는지 맥락이 약해진다. 정합성 맵이 먼저 엇박자를 설득해야 “그래서 보상을 어떻게 바꾸면 되는가”라는 다음 질문이 자연스럽게 발생한다.

## 2026-05-31: 정합성 맵을 5개 독립 텐션 축으로 전환

**컨텍스트**: 2D 화살표 기반 정합성 맵은 제도들이 흩어져 있다는 인상은 주지만, CEO가 각 축의 의미를 즉시 읽기 어렵다는 피드백이 있었다.

**결정 사항**:
- 결과 요약과 랜딩 미리보기의 핵심 시각화는 2D 화살표가 아니라 5개 영역별 독립 축으로 전환한다.
- 각 축은 `◆ 대표 철학`과 `● 실제 제도`의 거리로 텐션을 보여준다.
- 보상, 평가, 채용, 인력, 리더십은 각각 고유한 좌우 축 라벨을 가진다.
- 텐션이 큰 영역에는 도메인별 고정 템플릿으로 한 줄 비즈니스 리스크를 제공한다.
- 기존 `vectors` payload는 한 release 동안 호환용으로 유지하고, 새 `axes` payload를 병렬 추가한다.

**근거**: 정합성은 “하나의 추상 좌표에 흩어져 있음”보다 “대표가 정한 기준과 실제 제도 사이의 거리”로 보여줄 때 더 빠르게 이해된다. 또한 리스크 문구를 deterministic template로 두면 리포트의 신뢰성과 재현성이 유지된다.

## 2026-05-31: 대표 철학 입력을 별도 첫 단계로 분리

**컨텍스트**: 결과 화면에서 `◆ 대표 철학`이라는 표현을 쓰려면, 대표가 본인이 직접 기준을 설정했다는 인지가 먼저 형성되어야 한다.

**결정 사항**:

- 기존 `/diagnose/context`는 조직 규모, 성장 속도, 페인포인트 등 L1 문항만 담당한다.
- L0-4를 추가해 인력 안정성 축의 대표 철학 기준점을 만든다.
- 랜딩 CTA와 결과 화면의 진단 입력 복귀 경로는 새 첫 단계인 `/diagnose/philosophy`로 이동한다.

**근거**: 대표 철학이 조직 컨텍스트 안에 섞이면 결과 화면에서 “내 철학이 이거라고?”라는 의문이 생길 수 있다. 별도 첫 단계로 분리해야 정합성 텐션 맵의 기준점에 대한 오너십이 생긴다.

## 2026-05-31: 결과 요약 정합성 시각화를 텐션 축으로 교체

**컨텍스트**: 기존 2D 화살표 맵은 시각적으로 흩어짐은 보였지만, 각 영역이 어떤 기준에서 얼마나 어긋났는지 빠르게 읽기 어려웠다.

**결정 사항**:
- 결과 요약 최상단의 정합성 시각화를 `AlignmentTensionMap`으로 교체한다.
- 각 영역은 고유 축 위에서 인사 철학과 실제 제도의 거리를 보여준다.
- `tension >= 0.75`인 영역은 `엇박자`로 표시하고 한 줄 비즈니스 리스크를 함께 보여준다.
- backend `axes`가 없거나 stale 응답이면 frontend fallback axes를 생성해 화면이 비지 않게 한다.

**근거**: CEO가 첫 화면에서 “어느 영역이 내 지향점과 반대로 움직이는지”를 3초 안에 이해해야 Aha Moment가 생긴다. 1D tension row는 2D 좌표보다 해석 부담이 낮고, 이후 상세 분석과 트레이드오프 화면으로 이어지는 이유를 만든다.

## 2026-06-02: 철학 입력 직후 프로필 피드백 화면 추가

**컨텍스트**: 대표 철학을 별도 첫 단계로 분리했지만, 입력 직후 철학 자체의 충돌을 확인하는 화면이 없으면 결과 페이지에서 정합성 괴리를 처음 접하게 된다.

**결정 사항**:
- `/diagnose/philosophy/profile` 중간 화면을 추가해 L0-1~L0-4 선택을 보상, 리더십, 인재, 인력 기준으로 요약한다.

- 충돌이 있어도 재선택을 강요하지 않고, “이 모순을 인지한 상태에서 현행 제도 진단으로 이동”하는 흐름을 유지한다.

**근거**: Transition Gap은 정답을 강요하는 설문이 아니라 대표가 선택한 철학과 실제 제도 사이의 괴리를 드러내는 HITL 도구다. 철학 자체의 모순을 먼저 보여줘야 이후 제도 간 엇박자 분석이 자연스럽게 이어진다.

## 2026-06-02: 정합성 카드용 alignment_map 계약 확장

**컨텍스트**: 결과 요약을 카드 5장 구조로 바꾸려면 프론트가 좌표와 tension 값에서 라벨을 추측해서는 안 된다. 백엔드가 도메인 로직에 기반해 인사 철학, 현행 제도 방향, 정합 퍼센트, 상태 라벨을 명시해야 한다.

**결정 사항**:

- 기존 `philosophy_position`, `actual_position`, `tension`, `tension_level`, `business_risk`는 호환성과 분석 근거 보존을 위해 유지한다.

## 2026-06-03: 결과 요약 정합성 시각화를 카드형 괴리 분석으로 전환

**컨텍스트**: 슬라이더/축 방식은 거리 해석 부담이 크고, 결과 페이지 첫 화면에서 “어떤 제도가 어떤 방향으로 어긋나는지”를 바로 읽기 어렵다는 피드백이 있었다.

**결정 사항**:
- 결과 요약 최상단은 5개 카드로 구성하고, 각 카드에 인사 철학, 현행 제도, 정합 퍼센트, `일치/주의/심각` 상태를 표시한다.
- 숫자 거리값과 텐션 표현은 제거하고, `정합성 괴리`, `핵심 엇박자 요인` 언어로 통일한다.
- 카드 아래에는 `현행 인사제도 방향 요약`과 `제도 간 충돌 경고`를 배치한다.

**근거**: 대표/Head of People이 첫 화면에서 “내 철학과 실제 제도 중 어디가 어긋났는가”를 빠르게 읽어야 이후 상세 분석과 시나리오 비교로 이동할 이유가 생긴다.

## 2026-06-03: 채용·인력운영 진단 항목과 반복 진단 히스토리 보강

**컨텍스트**: 기존 인력운영/채용 화면은 이직률, 핵심 인재 이탈, 채용 소요 기간 중심이라 핵심 인재 기준, 대체 계획, 채용 브랜딩, 온보딩 추적 같은 운영 기반을 충분히 묻지 못했다. 또한 반복 진단 시 변화 추적 설계가 필요했다.

**결정 사항**:
- `2-1-4` 핵심 인재 식별 기준, `2-1-5` 핵심 포스트 대체 계획, `2-2-4` 채용 브랜딩 자산, `2-2-5` 온보딩 적응 추적 문항을 추가한다.
- `2-1-4`와 `2-2-5`는 가시성 기본 항목에 포함해 “측정 가능한 운영 기준”으로 관리한다.
- 결과 요약에는 브라우저 로컬 저장 기반의 최근 진단 히스토리를 추가해 정합성·가시성 변화와 이전 우선 영역을 보여준다.
- 배포 준비를 위해 프론트 API URL, 백엔드 CORS origin, frontend redirect를 환경 변수로 제어한다.

**근거**: Transition Gap은 제도 추천보다 “운영 기준이 실제로 작동하는가”를 먼저 보아야 한다. 핵심 인재와 채용 적응 데이터를 추가하면 성장 조직의 병목이 더 선명해지고, 반복 진단 기록은 8주 deploy에서 변화 신호를 보여주는 최소 장치가 된다.

## 2026-06-07: 기능 확정 단계의 보류/구현 기준 재정의

**컨텍스트**: Next.js + FastAPI 시안으로 전환된 뒤에도 루트 `AGENTS.md`에 과거 Streamlit 기준이 남아 있었고, PDF/브랜드/LLM/반복 진단 같은 보류 항목의 경계가 다음 작업 세션에서 혼선을 만들 수 있었다.

**결정 사항**:
- 현재 작업 기준을 `frontend/` Next.js와 `backend/` FastAPI로 고정하고, 과거 Streamlit 기준 문서는 `docs/legacy/`에 보존한다.
- 철학 충돌은 재선택을 유도하지 않고, 충돌을 인지한 상태로 진단을 계속 진행한다.
- PDF 생성과 정식 리포트 정보 구조는 기능 확정 이후 별도 단계로 보류하고, 현재는 `window.print()` 기반 인쇄/PDF 저장만 유지한다.
- 매트릭스에서 시나리오를 검토해 `/scenarios?scenario=...`로 넘기는 현재 흐름을 유지한다.
- 제도별 도입/보류/대체 검토는 시나리오 화면 안에서만 반영하고, 로드맵 자동 재계산이나 PDF 연결은 하지 않는다.
- 반복 진단과 데이터 저장은 localStorage 기반 상태와 SQLite 이벤트 로깅을 유지하고, 고객사별 workspace/DB는 유료 고객 확보 후 별도 스코프로 둔다.
- LLM/Gemini 연동은 현재 MVP에서 제외하고 규칙 기반 분석을 유지한다.

**근거**: 현재 단계의 목표는 기능 흐름을 안정화하는 것이며, PDF·브랜드·데이터 인프라·LLM은 각각 별도 설계 결정을 요구한다. 범위를 분리해야 MVP 데모 흐름이 꼬이지 않고 다음 세션에서도 같은 기준으로 작업할 수 있다.
